# 月度工作总结表格模板

最终输出必须使用中文 Markdown 表格，并严格遵守以下要求。

## 输出结构

最终输出分为两部分：

1. 主任务拆分表
2. `本月收获与反思` 附表

## 默认输入约束

- 当任务是“根据项目目标来源直接填写月总 / 月报 / 飞书总结表”时，默认至少提供：
  - `1 个项目目标来源`
  - `1 个输出目标`
- `项目目标来源` 优先指向包含 `目标 / 关键交付 / 每周行动 / 完成情况` 的周目标、周会文档或项目管理 Base
- `输出目标` 优先指向最终要落表的月总文档、月报表格或飞书文档中的指定章节 / 表格
- 在这个场景下，先按来源文档做字段映射，再结合 Dayflow / GitLab 做事实校准和工时归类

## 固定表头

```markdown
| 目标 | 关键成果（交付物/数据） | 关键行动举措（对齐组织目标拆解，需体现延期情况） | 完成情况（成效、问题、风险、措施） | 工作质量 | 工时/D（具体人天） | 自评难易（难/中/易） | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| ... | ... | ... | ... | ... | ... | ... | ... |
```

## 本月收获与反思附表

```markdown
## 本月收获与反思

| 类别 | 分享 |
| -- | -- |
| 收获/启发/成长 | ... |
| 反思/自我批评 | ... |
```

## 核心规则

- 字段必须作为表头横向展开，不要改成 `栏目 / 内容` 的纵向二列表
- 每一行代表一个任务拆分 / 工作流，而不是某一个字段说明
- 单元格内容较长时，使用 `<br>` 控制换行
- 如需列出多条内容，优先在单元格内使用 Markdown 无序列表风格组织明细，例如 `- ...<br>- ...`
- 尽量把每个条目写短，避免一个单元格里出现过长的大段描述
- Dayflow 可用时，工时与人天来自 Dayflow；GitLab 只用于补充交付与协作线索
- 未检测到 Dayflow 时，允许继续输出 GitLab-only 月报，但必须明确写出工时 / D 暂缺
- 无法直接证实的地方必须写成保守推断，不得虚构
- 如果同时提供了周目标 / 项目周会文档，默认按以下规则映射：
- 周目标中的目标 -> `目标`
- 周目标中的关键交付 -> `关键成果（交付物/数据）`
- 周目标中的每周行动 -> `关键行动举措（对齐组织目标拆解，需体现延期情况）`
- 周目标中的完成情况 -> `完成情况（成效、问题、风险、措施）` 的进展骨架
- 工时 -> `工时/D（具体人天）`，必须量化统计，并优先来自 Dayflow
- 主观评估 -> `工作质量` 与 `自评难易（难/中/易）`
- 同一批 Dayflow / GitLab 证据只能归属到一个任务拆分；如果存在交叉，必须先做互斥归类，避免重复累计工时
- 某个预研事项如果只有 GitLab 证据、缺少稳定的 Dayflow 独立工时，优先并入它直接支撑的正式交付项，不要单独伪造工时
- `本月收获与反思` 必须单独输出成第二张表，不要混进主任务拆分表的 `备注` 列
- 第二张表固定只有两行：
  - `收获/启发/成长`
  - `反思/自我批评`

## 字段填写要求

### 目标

- 第 1 条 bullet 重点写这一行任务拆分围绕哪个项目、哪类交付目标展开
- 第 2 条 bullet 重点写自己在该任务中获得了什么成长、沉淀了什么能力
- `目标` 仍然只占一列，但单元格内建议拆成 2 条 bullet 书写
- 不要显式写 `组织目标：...`、`个人目标：...` 这类标签
- 不要把两层目标压成同一条长句
- 优先使用“围绕 xx 项目推进 xx 交付”这类表达
- 允许基于 Dayflow / GitLab 轨迹做保守归纳，但表达要像评审材料，避免写成数据来源说明

### 关键成果（交付物/数据）

- 优先写可观察的交付动作：代码提交、MR、重构事项、文档、方案、评审、排障结果
- 如果来源是周目标文档，优先把“关键交付”映射成这一列；但表述时仍要用最终实际交付口径整理
- 不要把 Dayflow 卡片数、GitLab 事件数、工时统计等明细数据写进这一列
- 这一列只需要回答“最终交付了什么、推进了什么、沉淀了什么”
- 代表性事项建议控制在 2-4 个，并用更正式的交付口径表达

### 关键行动举措（对齐组织目标拆解，需体现延期情况）

- 写清楚围绕该任务拆分做了哪些关键动作
- 说明这些动作如何对齐组织目标
- 如果来源是周目标文档，优先把“每周行动”映射成这一列，再按 Dayflow / GitLab 实际轨迹补充关键动作
- 必须补一句延期情况：
  - 有证据时写明延期 / 阻塞 / 待跟进
  - 没有证据时写：`未从 Dayflow / GitLab 数据中看到明确延期证据。`

### 完成情况（成效、问题、风险、措施）

同一个单元格中至少覆盖这四项：

- 成效
- 问题
- 风险
- 措施

建议使用：`成效：...<br>问题：...<br>风险：...<br>措施：...`

- 这一列必须是对 Dayflow 与 GitLab 证据的综合总结
- 如果来源是周目标文档，优先把“完成情况”映射成这一列的进展骨架
- 这一列中引用的完成时间、推进节奏和阶段判断，必须再用 Dayflow / GitLab 的真实时间窗口校准，不要直接把周目标中的计划日期写成实际完成时间
- 不要分别解释“Dayflow 做了什么、GitLab 做了什么”
- 语言要适合直接发给团队评审，尽量写成阶段结果判断和管理视角总结

### 工作质量

- 结合连续投入、闭环情况、代表性交付、是否存在明显返工来判断
- 这一列是主观质量评价，建议用 1 句短评写清“质量是否稳定、闭环是否扎实”
- 避免空泛评价，尽量给出基于数据的短句

### 工时/D（具体人天）

- Dayflow 可用时，必须同时展示小时与 D
- 工时必须优先使用 Dayflow 可识别工作窗口统计；GitLab 仅用于校准归属，不用于补工时
- 公式固定为：`D = 小时 / 8`
- 保留 2 位小数
- 未检测到 Dayflow 时，明确写成“暂无法折算”之类的缺失说明，不要伪造 `0.00`

### 自评难易（难/中/易）

- 给出 `难 / 中 / 易` 中的一个结论
- 后面补一句简短说明，解释为什么这么判断

### 备注

- 用于说明数据边界、推断来源、证据不足项、系统覆盖限制等

## 本月收获与反思填写要求

### 收获/启发/成长

- 基于本月投入时间最多的主题、典型交付动作、MR 闭环情况做归纳
- 可以写能力成长、方法沉淀、协作启发、流程优化体会
- 优先使用“从本月活动轨迹看”“结合 Dayflow 与 GitLab 记录”这类表述

### 反思/自我批评

- 优先写有证据支撑的问题，例如：
  - 上下文切换多
  - 闭环证据不足
  - 阻塞/待跟进信号较多
  - 缺少 journal 记录导致目标与反思沉淀不足
- 建议同时写出下一步改进方向，但不要展开成大段行动计划
